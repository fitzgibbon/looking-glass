#+TITLE: looking-glass: profunctor lenses in emacs lisp
* Features

** Scope and goals

- Build a profunctor-based optics library for Emacs Lisp: `looking-glass` (`lg`).
- Target practical parity with PureScript `profunctor-lenses` and key pieces of Haskell `lens`.
- Prefer Emacs Lisp ergonomics over direct 1:1 API porting.
- Start with core optics and built-in Emacs Lisp data structures.
- Add richer domain packages (JSON, YAML, TOML, buffers) later in this repository.

** Core design principles

- Use profunctor encodings for optic composition and interpretation.
- Keep the optic representation purely profunctor-based (no ad-hoc alternate encodings in core).
- Optimize for extensibility: new optic kinds should be definable as thin wrappers over profunctor constraints.
- Keep APIs explicit and discoverable (`lg-` prefix, consistent verb naming).
- Provide both composable optic values and convenience operation functions.
- Favor immutable-style update semantics while remaining idiomatic for Elisp users.
- Ensure optics are lawful where meaningful and document edge-case behavior.

** Core optic types (v1)

- Isomorphism (`iso`)
- Lens (`lens`)
- Prism (`prism`)
- Traversal (`traversal`)
- Setter / Getter / Fold-style derived optics
- Indexed variants for list/vector/string positions and map-like keys

** Optic extensibility model

- Define one canonical internal optic representation and composition path.
- Expose extension points for defining new optic types via profunctor constraints and interpreters.
- Keep constructors separate from interpreters so new optics can reuse existing operations.
- Require every new optic type to document its laws and composition behavior.

** Core operations (v1)

- `view` / `preview` / `to-list-of`
- `over` / `set`
- `has` / `isn\='t`-style predicates (or Elisp-idiomatic equivalents)
- Composition operator/function (`compose`, threaded helper macros)
- Optional state-style helpers if they fit Elisp ergonomics

** Built-in data structure support (initial package)

- Pairs and cons cells (`car`, `cdr`, nested pair paths)
- Lists (head/tail/indexed element traversals)
- Plists (key lens/prism, optional missing-key semantics)
- Alists (key optics with configurable equality)
- Vectors and arrays
- Strings (character index traversal/lens, substring optics if feasible)
- Hash tables (key optics as optional v1.1 target)

** Ergonomics and compatibility

- Emacs version support policy (define minimum supported version).
- Clear behavior for nil vs empty values across structures.
- Distinguish nil-as-missing from nil-as-value in APIs that touch associative structures.
- Optics should preserve focus semantics: any nil produced by the focus is passed through unchanged.
- Where the optic kind allows, provide non-nil guarantees for total optics (and document when guarantees do not apply).
- Customization points: equality predicates, missing-key behavior, defaults.
- Optional macros to reduce ceremony in common composition/update flows.

** Quality and docs

- ERT test suite with law-inspired tests and behavior tests.
- Examples-first README and per-optic docstrings.
- Benchmarks for core operations on typical data structures.
- Changelog and migration notes for API changes.

** JSON optics package plan (`looking-glass-json`)

- Do not force JSON through plists as the single internal representation.
- Use a JSON-native optic layer over parsed Elisp structures with configurable object mapping.
- Support object representations produced by `json-parse-string` / `json-read`:
  - plist-backed objects
  - alist-backed objects
  - hash-table-backed objects
- Define core JSON optics:
  - `j-object-key` (affine traversal/prism-like key focus)
  - `j-array-index` (affine traversal for array positions)
  - `j-values` (traversal over object values)
  - `j-members` (indexed traversal over key/value pairs)
  - scalar prisms: `j-string`, `j-number`, `j-bool`, `j-null`
- Define explicit null semantics:
  - JSON null represented by a dedicated sentinel (not bare nil)
  - keep missing vs null vs value disambiguated in previews
- Provide parser/renderer optics:
  - parse prism: JSON string -> JSON value
  - review: JSON value -> JSON string
  - options for object type, array type, null sentinel, key type
- Add path composition helpers:
  - key/index chaining with predictable failure behavior
  - optional path parser for JSON Pointer style paths in a later phase
- Testing plan:
  - roundtrip parse/review tests
  - missing/null/value disambiguation tests
  - representation parity tests (plist/alist/hash-table backends)
  - indexed member traversal tests
