* Functor-Centric Comparison: Haskell lens, PureScript profunctor-lenses, and looking-glass

This document explains the libraries from a *functor interpreter* point of view:

- an optic is one polymorphic program
- operations come from choosing a functor/applicative/profunctor interpreter
- concrete helpers like =view=, =over=, =preview=, =review= are interpretations.

That model is true in Haskell, PureScript, and =looking-glass=; the biggest differences are dynamic runtime checks and Elisp data shapes.

* Core idea by interpreter

| Operation family | Typical interpreter idea | Haskell/PureScript name | looking-glass |
|-
| update | identity effect | =over= / =set= | =lg-over= / =lg-set= |
| collect foci | constant accumulation | =toListOf=, =foldMapOf= | =lg-to-list-of=, =lg-fold-map-of= |
| preview first match | short-circuit optional accumulation | =preview= | =lg-preview= |
| construct | tagged output direction | =review= | =lg-review= |
| indexed update/fold | indexed interpreter + accumulation/identity | =iover=, =ifoldMapOf= | =lg-iover=, =lg-ifold-map-of= |

In =looking-glass= this is explicit in architecture:

- plain optics are interpreted with star-like profunctor runners
- folds/getters use accumulation interpreters
- reviews use =Tagged=-style interpretation
- reversal uses =Re=-style interpretation
- indexed optics use an explicit indexed wrapper (=lg-indexed=).

* Constructor correspondence

| Concept | Haskell lens | PureScript profunctor-lenses | looking-glass |
|-
| iso constructor | =iso= | =iso= | =lg-iso= |
| lens constructor | =lens= | =lens= | =lg-lens= |
| prism constructor | =prism= / =prism'= | =prism= / =prism'= | =lg-prism= |
| traversal constructor | traversal/wander builders | wander-style constructors | =lg-traversal= |
| affine traversal | dedicated affine modules/combinators | =Data.Lens.AffineTraversal= | =lg-affine= |
| grate | =grate= | =grate= | =lg-grate= |
| review constructor | =unto= / =review= plumbing | =unto= / =review= plumbing | =lg-unto= |

* Composition model

=looking-glass= now keeps one explicit composition API:

- =lg-compose= for plain optics
- =lg-compose-indexed= for indexed optics

Ordering is explicit and functional: =(lg-compose o1 o2 o3)= means run =o3= then =o2= then =o1=.

* Indexed model (functor-centric)

Haskell/PureScript carry indexing via indexed profunctor/typeclass structure.

=looking-glass= mirrors this with explicit runtime representation:

- indexed payloads run through =lg-indexed=
- lifted non-indexed paths use =lg-no-index= sentinel
- indexed folds/updates (=lg-iover=, =lg-ito-list-of=, =lg-ifold-map-of=) are direct indexed interpretations.

* Reversal and dual classes

Haskell/PureScript profunctor optics commonly model reversal with =Re= plus dual classes:

- =Strong <-> Costrong=
- =Choice <-> Cochoice=

=looking-glass= now follows that shape for reversible paths by adding dual operations to profunctor interpreters:

- forward: =first=, =right=
- dual: =unfirst=, =unright=

This enables broader =lg-re= support than dimap-only reversal, including lens/prism chains built from those structures.

** Practical instances in this codebase

- function profunctor ("->"-style): supports =first/right= and =unfirst/unright=
- forgetful profunctor ("Forget"-style): supports =first/right= and =unfirst/unright=

These are directly analogous to practical instances you see in Haskell/PureScript optics internals.

** Real reversal example (long prism+lens chain)

#+begin_src elisp
(setq id-optic
      (lg-compose
       lg-right-o
       lg-cdr
       lg-car
       lg-number-string))

(setq response '(right nil "42"))

(lg-view id-optic response)
;; => 42

(lg-view (lg-re id-optic) 123)
;; => (right nil "123")

(equal (lg-view (lg-re id-optic) 42) response)
;; => t
#+end_src

This is the user-visible effect of =Costrong/Cochoice=-style support in reversal.

** What reversal still does not cover

As in mainstream Haskell/PureScript practice, traversal-style =wander= paths are not generally reversible via dual machinery alone.
So =lg-re= remains strongest for iso/lens/prism/review-style optics.

* Runtime vs static guarantees

Functor semantics are aligned, but enforcement differs:

- Haskell/PureScript: many constraints proven statically
- Emacs Lisp: constraints checked at runtime

So the *meaning* of interpreter-based operations matches, but failure mode is runtime signaling in =looking-glass=.

* Data-model adaptation

Compared with typed ADTs/records, =looking-glass= targets Elisp-native containers:

- cons cells/lists
- plists/alists
- hash tables
- vectors/strings/bool-vectors

This mainly affects container dispatch for =lg-ix= / =lg-at= / =lg-nth= extension points, not the interpreter model itself.

* Nil and optional focus semantics

In Elisp, =nil= is both false/empty-list and valid payload, so =looking-glass= uses tagged maybe results for disambiguation:

- missing focus: =lg-nothing=
- present focus: =(lg-just . VALUE)= (where =VALUE= can itself be =nil=)

This keeps preview/fold semantics stable under the functor-centric interpretation model.

* Practical takeaway

If you already think in =Functor=/=Applicative= optics terms from Haskell or PureScript, =looking-glass= is conceptually the same game:

- define optics once
- choose interpreter to get operation behavior
- adapt only for dynamic checks + Elisp container semantics.
