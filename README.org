#+TITLE: looking-glass

Compositional profunctor optics for Emacs Lisp.

~looking-glass~ gives you one language for reading, transforming, and reviewing data across lists, vectors, plists/alists/hash tables, strings, JSON text, and regex matches.

- Express nested edits once, then reuse safely.
- Stay fully pure: transforms return new values, so composition stays predictable.

* Quick examples

** Promote names inside JSON text (no parser boilerplate)

#+begin_src emacs-lisp
(require 'looking-glass)

(let ((json "{\"user\":{\"name\":\"Ada\"},\"active\":true}"))
  (lg-over (lg-compose lg-json-parse
                       (lg-ix "user")
                       (lg-ix "name"))
           #'upcase
           json))
;; => "{\"user\":{\"name\":\"ADA\"},\"active\":true}"
#+end_src

#+RESULTS:
: {"user":{"name":"ADA"},"active":true}

** Bulk rewrite only capture groups

#+begin_src emacs-lisp
(require 'looking-glass)
(require 'looking-glass-regex)

(lg-over (lg-compose (lg-regex-all-groups "id=\\([0-9]+\\)" 1)
                     lg-number-string)
         (apply-partially #'+ 100)
         "id=1 id=22 id=333")
;; => "id=101 id=122 id=433"
#+end_src

#+RESULTS:
: id=101 id=122 id=433

** Insert and replace list slices

#+begin_src emacs-lisp
(require 'looking-glass)

(let ((numbers '(1 2 3)))
  (list
   (lg-set (lg-slice 1 1) '(4) numbers)
   (lg-over (lg-slice 1 3)
            (lambda (segment)
              (mapcar (lambda (n) (* n 10)) segment))
            numbers)))
;; => ((1 4 2 3) (1 20 30))
#+end_src

#+RESULTS:
: ((1 4 2 3) (1 20 30))

* Package map

- ~looking-glass.el~: pure optic core, composition, folds, indexed optics, reviews, and law-aligned primitives.
- ~looking-glass-regex.el~: pure regex and text traversals (~lg-regex-all~, ~lg-lined~, ~lg-utf8~).
- ~laws.org~: law expectations and nil-disambiguation notes.

* MELPA manual publishing

MELPA packages repository contents from the selected branch/commit. It does not build manuals from Org/Texinfo build artifacts.

This repository uses CI to build manuals and a dedicated release branch strategy:

- source manuals live in ~doc/*.org~;
- CI builds ~.texi~, ~.info~, and ~dir~ with ~scripts/build-manuals.sh~;
- release publishing targets a separate ~melpa-release~ branch (dry-run workflow currently reports the exact commands only);
- release dry-run steps are gated on a successful full test run.

For ~M-x info-display-manual~ support in installed packages:

- package contents must include the generated ~*.info~ files;
- package root must include a ~dir~ file so package activation can add the directory to ~Info-directory-list~.
